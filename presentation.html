Documentación Detallada de Modelos de Datos y Entidad-Relación

1. Base de Datos Principal Persistente (PostgreSQL)

¿Por qué usamos PostgreSQL y qué lugar ocupa?

Justificación Arquitectónica: PostgreSQL es una base de datos relacional que garantiza el cumplimiento ACID (Atomicidad, Consistencia, Aislamiento y Durabilidad). Se utiliza como la fuente de la verdad a largo plazo. Los datos como credenciales de usuarios, perfiles de personajes y el registro histórico de quién ganó qué partida no pueden permitirse inconsistencias.

Uso en el Sistema: El DB Server (Java 25 + Spring Boot) es el único microservicio que se comunica con esta base de datos. Ninguna lógica del juego corre aquí.

La tabla clave para la resiliencia: game_state_dumps. Para evitar que una caída del motor de juego destruya partidas de horas de duración, el Middle Server envía una copia exacta de su memoria cada ~15 minutos. Si el servidor se apaga, al reiniciar lee el state_json de Postgres y retoma la partida exactamente donde se quedó.

2. Base de Datos Analítica e Histórica (MongoDB)

¿Por qué usamos MongoDB y qué lugar ocupa?

Justificación Arquitectónica: MongoDB es una base de datos NoSQL orientada a documentos. Se ha elegido porque los datos estadísticos e históricos crecen exponencialmente y no requieren integridad referencial estricta. Guardar objetos JSON masivos y altamente anidados (como un "snapshot" entero de una partida con todos sus ejércitos) es extremadamente ineficiente en SQL, pero natural y muy rápido en MongoDB.

Uso en el Sistema: Extraer estadísticas masivas ("¿Qué clan tiene mejor Win-Rate?") sobre millones de combates paralizaría a PostgreSQL. Al separar estas analíticas en MongoDB, protegemos el rendimiento del sistema de login y gestión de usuarios. El Middle Server envía estos informes al DB Server en segundo plano (las batallas al ocurrir, y los snapshots cada 2 horas).

3. Modelo de Estado En Vivo (Memoria - Middle Server)

¿Por qué vive en la Memoria y cómo funciona el Motor?

Justificación Arquitectónica: Viking Clan Wars es un juego de estrategia en tiempo real. Si cada movimiento de tropas, cálculo de daño, o recolección de oro requiriera hacer una consulta de escritura (INSERT/UPDATE) en una base de datos en disco, el juego sufriría de un "lag" injugable y el servidor colapsaría por saturación de I/O.

Uso en el Sistema: Por ello, el Middle Server (Node.js) carga y mantiene toda la partida viva exclusivamente en la Memoria RAM. Las peticiones a través de WebSockets (Socket.IO) modifican las propiedades de estos objetos en milisegundos, retransmitiendo el nuevo estado a los clientes conectados.

La Rueda del Tiempo (Time Wheel / eventQueue): Para evitar tener miles de temporizadores setTimeout corriendo a la vez (lo que desestabiliza Node.js), la arquitectura usa un patrón de Rueda del Tiempo. Todos los eventos futuros (como el impacto de un ataque) se guardan como un GameEvent en una cola MinHeap ordenada por timestamp. Un único bucle global lee la cima de la cola un par de veces por segundo y ejecuta el evento solo cuando llega su hora cronológica exacta. Esto garantiza que ningún evento ocurra fuera de orden.
